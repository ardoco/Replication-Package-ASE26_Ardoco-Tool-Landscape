# Nullness and JSpecify

ARDoCo adopts [JSpecify](https://jspecify.dev/) to make null-safety explicit and consistent across the codebase, improving IDE analysis without changing runtime behavior.

## Core Rules
- Default is non-null: Packages are marked `@NullMarked` during the build.
- Opt-in nulls only: Use `org.jspecify.annotations.Nullable` where a reference can be null (fields, parameters, return types).
- Single annotation source: Do not use other `@Nullable` types (e.g., `javax.annotation.Nullable`, `org.eclipse.jdt.annotation.Nullable`).
- No package-info commits: `package-info.java` files are generated by the build and are ignored by VCS.

## Examples
```java
import org.jspecify.annotations.Nullable;

class UserService {
    // Field that may be absent
    private @Nullable String nickname;

    // Parameter can be null, return is non-null by default
    String greet(@Nullable String name) {
        var effective = (name == null || name.isBlank()) ? "there" : name;
        return "Hello, " + effective + "!";
    }

    // Method result may be null
    @Nullable String findDisplayName(String userId) { /* ... */ return null; }
}
```



## Build & Generated Sources
- The `jspecify-maven-plugin` generates `@NullMarked` `package-info.java` files during `process-sources` into the module source directory.
- `package-info.java` is intentionally listed in `.gitignore`. Do not add or commit these files manually.

Top-level configuration snippet (already present):
```xml
<plugin>
  <groupId>org.fuchss</groupId>
  <artifactId>jspecify-maven-plugin</artifactId>
  <version>0.1.0</version>
  <executions>
    <execution>
      <id>generate-jspecify-package-infos</id>
      <goals><goal>generate</goal></goals>
      <phase>process-sources</phase>
      <configuration>
        <generatedSourcesDirectory>${project.build.sourceDirectory}</generatedSourcesDirectory>
      </configuration>
    </execution>
  </executions>
</plugin>
```

## Adding New Modules
- Ensure the `jspecify-maven-plugin` is applied (inherited from the parent). If you create a new module with its own `pom.xml`, include the plugin (no version needed if managed in the parent).
- Verify `org.jspecify:jspecify:1.0.0` is available via the parent dependencyManagement; if not, declare it as a dependency.

## IDE Tips
- IntelliJ IDEA and Eclipse provide JSpecify-aware null analysis. Make sure inspections for nullability are enabled.
- VS Code: Use the Java extensions; most IDE hints work once dependencies are resolved.

## Conventions
- Prefer non-null returns and parameters; use `@Nullable` only when necessary and clearly documented in Javadoc.
- Keep null checks explicit near boundaries (I/O, deserialization) and convert to non-null invariants quickly.
- When overriding methods, respect and preserve the nullness contract.
